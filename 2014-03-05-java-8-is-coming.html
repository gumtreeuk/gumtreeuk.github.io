Java 8 is coming

By Pere Villega
Tweet

It's March 2014 and, at last, it is happening. Barring any last minute surprise, Java 8 will be released this month after 3 years of delays and spec changes. Java 7 was released on 2011 and some of the promised functionality in that version was moved to Java 8. In that note, this release doesn't start too well with more features being dropped, mainly Project Jigsaw which adds a module system to the Java platform and was initially scheduled for Java 7.

Personally, I'm not a big fan of this release. Given what you can currently do with alternate JVM languages like Scala or Clojure, not to mention Kotlin and others, I feel that it is too little too late. People are excited about lambda expressions but, given the rise (and benefits) of functional programming, it feels like an opportunity has been missed.

The fact is that Java 8 has arrived, so we can as well see what's new in this release.

Nashorn

Let's start with Javascript. JVM had Rhino), a Javascript engine to run Javascript in the JVM. Nashorn) is a new implementation that takes advantage of new Java 8 byte code and is potentially much faster. It even includes a command line tool (jjs) to run scripts directly.

Personally, I'll stay with Scala-js.

New Date and Time API

Let's face it, the current Date/Time API in Java is worse than coding in VB. Joda-Time is the de-facto library to work with dates and time for a reason, not only its quality but also because of the native API being so bad. Luckily, Java 8 introduces some changes in the area which may turn using Joda-Time something optional.

Don't take it wrong, I like Joda-Time. But I believe that every additional dependency is one more point of failure introduced in the application. Reliable as they may be, more moving parts mean more potential sources of error. I like simplicity, and using a native API included by the language (if it is not vastly inferior) is, in my book, the correct choice.

There are 4 main classes in the new API:

  • LocalDate: stores a date with day, month and year. No Timezone information.
  • LocalTime: displays the current time. No Timezone information.
  • LocalDateTime: displays both Date and Time. No Timezone information.
  • ZonedDateTime: displays Date and Time and contains Timezone information.

Working with the API is simpler, too. For example, to modify a stored time object you do:

LocalTime current = LocalTime.now();
LocalTime CoB = current.plus(9, HOURS);

Another neat addition is the introduction of Clock, a class that provides access to the current time including Timezone information. A replacement for System.currentTimeMillis() that may facilitate working with timed tests due to its integration with DateTime API.

One example of its use:

Clock.systemDefaultZone();

Optional

In Java 8 Optional is a standard class available in java.util. For those not familiar with the construct, Optional is a way to avoid propagating null values across the application. Optional uses generics and can have two values: empty or non-empty. In scenarios where you may or may not get a result, you return an Optional of a type and work around the presence of the element.

For example:

Optional<String> opt = toBeOrNotToBe();

if(opt.isPresent())    
  return opt.get();
else
  return "";

You may argue that using null gives similar code. The advantage of Optional is not saving keystrokes, but making the possibility of an empty value explicit. Instead of having to check for nulls every time you call a method you can rely that a method that doesn't implement Optional will return a value, even if that value is an empty list. Also, Optional provides some collection-like methods, such as filter or map, which facilitates treating Optional as a collection and merging it in stream processing.

Those used to work with Optional in Scala or by using Guava will be glad of this change. Hopefully this will propagate soon to libraries, making them easier to use and reducing the amount of Null Pointer Exceptions in our applications.

Annotations on Java Types

Annotations are ubiquitous in Java. Love them or hate them, they are used in most frameworks. Java 8 allows you to use annotations in several new places where you couldn't before. Rumour has it that this will allow frameworks to check for common errors like null values, race conditions or some other tests. One example is Checker, a type system enhancer for Java.

Let's see some self-explanatory examples:

// Type cast
mustExist = (@NotNull Integer) choice;

// Object creation
new @NoSQLInjection LittleBobbyTables(String input);

// Adding interfaces
class MySetWrapper<T> implements @Immutable Set<@Immutable T> 

// Declaring exceptions
void crossTheBridge() throws @RunYouFools BalrogException

Now, call me biased but if you can't use Scala I'd rather suggest you to look at Guava before getting too excited with these annotations. Right now the aim of Checker and similar tools can be achieved via constructs like Optional, Preconditions or Immutable Collections. Annotations may provide similar behaviour but are harder to track by a developer using your code, while the aforementioned alternatives are more intuitive and visible wherever that code is used.

That said, I'm pretty sure having this functionality may allow new frameworks to take advantage of it and provide very useful behaviour. But, as far as I'm aware, we are not there yet.

Lambda Expressions

We get to the big feature of Java 8. Developed as Project Lambda. A Lambda Expression) is an anonymous function that can be defined and called "on the fly", capable capturing values of variables in scope when defined. They are extremely useful when passing arguments to higher-order functions or for some other functional operations.

A typical example is a lambda expression used to sort an Array by length of its elements:

Arrays.sort(words, 
(String w1, String w2) -> { w2.length() - w1.length() });

As you can see in the example lambdas reduce clutter. Instead of declaring a Comparator class and adding all that boilerplate, we provide the criteria to use as an argument. Note that we could store that lambda in a variable.

Alternatively, you can pass existing methods instead of a lambda. Through Method References (syntax: ::) you can access a method in a class an use it as the parameter. You can use for this purpose static methods, instance methods and constructors. Basically, almost anything you can declare in Java. For example, to use println as a lambda:

myList.forEach(System.out::println);

The snippet above will print all elements of the list to System.out.

Functional Interfaces

Functional interfaces are interfaces with exactly one non-default method. The package java.util.function declares several functional interfaces but you can define your own, for example:

@FunctionalInterface
public interface MyFunctionalInterface {

  public int theQuestion();

}

The annotation FunctionalInterface indicates to the compiler that this is a functional interface. They behave like interfaces and can extend other interfaces assuming this will not break the contract of having only one non-default method.

From the point of view of the compiler the interface defines a template for a lambda. This means that the type of a functional interface can be used as the type for a lambda. For example:

MyFunctionalInterface deepThought = () -> return 42; 

public void answerMe(MyFunctionalInteface calculator) { ... }

answerMe(deepThought);
answerMe(() -> return 42;)

In the snippet above we first see a lambda stored as a variable whose type is a functional interface. Next we see a method, doSomething, that accepts a functional interface as a parameter. The following two lines show two valid ways of using the method, either by providing an instance of the functional interface or by passing a valid lambda that matches the definition of the interface as a parameter.

Default methods

You may have heard of Default Methods as Defender Methods or Virtual Extensions. You can think of them as a way to turn interfaces into mixins. You can add as many default methods as you want to an interface and, if not overridden, the default implementation in the interface will be used. You can also add them to functional interfaces: remember that default methods don't count towards the limit in this type of interfaces.

Be aware that if a class implements two default methods with the same name but belonging to different interfaces you will get a compiler error. This helps avoiding issues like diamond problem.

An example:

public interface MyInterface {
  default public int rollD6() {
    return 4; //guaranteed random by roll of dice
  }
}

Similarly, now you can add static methods to your interfaces:

public interface MyInterface {

  default public int rollD20() {
    return Constants.thaco;
  }
}

This may be the most useful addition to Java in this release, even more than lambdas. Lambdas are something akin to a patch as Java is not a functional language. On the other hand default methods help developers favour composition over inheritance by providing the necessary tools. This can have a direct impact over code stability and refactoring while, at the same time, removing the main drawback of having to reimplement all methods of a given interface in each implementer.

Collections enhancements

A couple of handy enhancements have been added to Java collections, all of them available by default. The first one is the addition of functions map, filter and reduce to the standard collections. You could have access to then in Java 7 via Guava, but having them in the standard API is an improvement.

For example, we could read a stream of posts and add the number of comments on each one to see how many comments we have in total:

reddit.stream()
  .map(post -> if(post.isCatPost) return post.getComments(); else return 0;)
  .reduce(0.0, (c1, c2) -> c1 + c2);

From the point of view of a Scala developer, this is another proof that Java is playing catch-up with modern languages. This was a necessary addition to simplify working with collections, but again it fails short when you see there is no fold or any of the other helper methods commonly used in other languages.

Another enhancement to collections is the Stream interface. Many core library classes can now return a stream of its contents, either in a sequential or parallel way. The stream is populated lazily, which facilitates working with large collections of data. They support sorting and other methods like limit. And, of course, you can create your own stream. We can, for example, create a stream from a list of numbers, sort it and get the first 10 elements:

Stream.of(aListOfNumbers)
    .sorted() 
    .limit(10)

Streams work well with Collectors, a set of classes intended to amalgamate the contents of a stream into a unique result, be it a collection or a result. You can use a default collector to obtain the average of the values in a stream, as seen in the official documentation: 

Map<Person.Sex, Double> averageAgeByGender = roster
    .stream()
    .collect(
        Collectors.groupingBy(
            Person::getGender,                      
            Collectors.averagingInt(Person::getAge)));

That's it

That's the list. Some interesting additions that will benefit software stability and some half-hearted changes. If you are already using a more modern language you will hardly go back, but they are steps in the right direction. Now, the countdown for Java 9 starts: Currency API, Project Jigsaw and more. In the meantime, enjoy your shiny JVM.
