Optimising JavaScript performances with asynchronous loops

By Paolo Tozzo
Tweet

At Gumtree we want to offer the best user experience to our customers, and we believe that one of the first steps is not to make the user wait for something. Performance is definitely a feature, and we want to provide the fastest user experience, both during and after the page has been loaded.

What does that mean exactly? It means that while often most of the focus in terms of performance optimisation is on the page load, by reducing the number of requests and the page weight, we forget what happens after the page has been loaded. When our UI is being shown to the user, the browser can end up doing more work than it is supposed to, and this means a sloppy interface that could definitely have an impact on the user engagement.

And with the Mobile traffic expanding rapidly this is even more important.

JavaScript Performance

Now, performance problems are rarely caused by raw Javascript, far more often is the interacting with the DOM through Javascript that causes the issue.

However, especially when dealing with complex UIs or data structures, avoiding long running scripts is of paramount importance.

Loops in JavaScript

Loops are often a performance bottleneck in Javascript applications. That's because loop iterations happen in a synchronous way, so, the more iterations you have, the more it'll take to execute your code. If the loop is doing too much (especially updating the DOM) the risk is to freeze the UI, with the risk even to display the dreaded 'unresponsive script' browser dialog pop-up.

If you don't need your loop to be executed synchronously, and the order in which the elements of your collection are processed is not important, splitting the collection analysis into smaller and more manageable chunks can make a HUGE difference in terms of perceived performance of your UI.

Today we are going to use Javascript timers in order to free up the browser from the risk of long-running loop. This pattern is explained by Nicholas C. Zakas in his book Professional Javascript for Web Developers in the chapter on Advanced techniques.

An Important Consideration Before Starting

Before using the pattern I'm going to show you have to ask yourself how much is happening inside the loop. The more work you do outside the loop, the better is. A lot of time it's a poor design that is causing the problem. If you've done as much as you could to optimise your loop, but you still need to handle a large volume of data in the browser, and that is causing you some headache, this article will help you.

Let's code!

Let's have a look at a typical loop

for (var i = 0; i < collection.length; i++) {
    //do something
}

Now, if after your loop has been executed there is nothing that needs to deal with its outcome, and each iteration is treated separately from the others, it means that your loop can run asynchronously.

First, let's create the helper we need

function splitLoop(items, process, context) {
    var todo = items.concat(); //create a clone of the original
    setTimeout(function () {
        do {
            process.call(context, todo.shift());
        } while (todo.length > 0);
            setTimeout(arguments.callee, 25);
    }, 25);
}

First of all, we clone the original array. This shouldn't be always needed, but since the collection is passed by reference we want to be safe that we are not altering it, so we are going to work with the clone.

Then we use the setTimeout function to delay the execution of each iteration.

On each iteration we use Array's shift method in order to remove the first element of the collection and we carry on until we processed all items. We use do...while in order to execute the loop at least once. Then provide a context in order to set the value of this inside our loop.

Let's see now how we can use it with a test case.

// test data
var items = [];
for (var i = 0; i < 1000; i++) items[i] = i;

splitLoop(items, function (item) {
    console.log(item)
}, this);

That was easy... and of course, not really useful!

Let's now say we want to process a collection of dom elements and we are using jQuery.

splitLoop(  $(".items").toArray(),
            function (item) {
                $(item)
                    .processItem();
            },
            this
);

Presto! We converted the the object jquery is returning to an array, and we are now able to execute dom operations asynchronously.

Since we are working with asynchronous programming you need a callback function to pass since you don't know the point at which it will complete. Let's improve our helper!

function splitLoop(items, process, context, callback) {
    var todo = items.concat(); //create a clone of the original

    setTimeout(function () {
        var start = +new Date();

        do {
            process.call(context, todo.shift());

        } while (todo.length > 0);

        if (todo.length > 0) {
            setTimeout(arguments.callee, 25);
        } else {
            callback(items);
        }
    }, 25);
};

On each iteration now we check if we've reached the end of the collection, and when this happens the callback function is executed. Let's see it in action with the following code

// test data
function todo(item) {
    console.log(item);
};

function done() {
    console.log('done');
}

var items = [];
for (var i = 0; i < 1000; i++) items[i] = i;

splitLoop(items, function (item) {
        todo(item);
    }, this, done);

A further improvement could be to make sure that each chunk won't take longer than a specific amount of milliseconds.

function splitLoop(items, process, context, callback) {
    var todo = items.concat(); //create a clone of the original

    setTimeout(function () {

        var start = +new Date();

        do {
            process.call(context, todo.shift());

        } while (todo.length > 0 && (+new Date() - start < 50));

        if (todo.length > 0) {
            setTimeout(arguments.callee, 25);
        } else {
            callback(items);
        }
    }, 25);
}

Now we check than each iteration lasts less than 50 milliseconds.

Feel free to alter both 25 and 50 milliseconds, or if you prefer, to pass them as a further arguments for the function, I've never needed to change them, so I prefer to hard code them.

I hope you'll find the article helpful, feel free to comment and let us know any suggestion to further improve the technique.
